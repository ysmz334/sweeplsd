2. Across image scales

ELSED's defaults are tuned for 640×480 inputs, where its published speed leadership was established, so the same 150 photographs were Lanczos-downsampled and re-timed (same in-process harness, so compare ratios):

SweepLSD is ~2× faster than ELSED at every scale, including ELSED's native one.

Latency. Instrumenting the labeller's emission point over the whole corpus (222,507 segments): a segment is emitted a median 6.5 rows after its last pixel enters the detector (95% within 7.0, 99% within 7.5, worst 11.3) — ≈0.19 ms at 1080p30 video timing, independent of image height. Any detector that must hold the image has a latency floor of one frame period (33 ms) plus its processing time.

4. Where the time goes

Per-stage profile (sweeplsd_profile, median of 9 runs, one Full-HD photo, multi-pass driver so stages are separable):

stagemsshare
1. gaussian2.111%
2. gradient2.212%
3. edge (threshold + NMS)1.69%
4. endpoint candidates3.016%
5. labelling + judgment9.652%

Stages 1–4 are the auto-vectorized row kernels; stage 5 is the streaming labeller (inherently more scalar — per-pixel label bookkeeping). The one-pass driver runs the same kernels but keeps only a few rows of each intermediate, so its working set stays in L1/L2 — which is why it is faster than the multi-pass driver despite doing identical work.

5. Compiler dependence (measured honestly)

The kernels contain no SIMD intrinsics by design; the speed comes from compiler auto-vectorization.

compilerFull-HD detectnote
GCC (MinGW-w64 8.1)15–18 msfully vectorizes the byte kernels
MSVC 2022~53 mscorrect (all tests pass) but does not vectorize the byte kernels — prefer MinGW-w64 or clang-cl on Windows
Reproduce it. cmake -DSWEEPLSD_BUILD_BENCH=ON fetches the baseline detectors at configure time (LSD is AGPL-3.0 and is deliberately not vendored in this MIT repository), then run sweeplsd_time_methods <dir-of-pngs>. Genuine-EDLines timings need OpenCV (sweeplsd_edlines_runner is built automatically when OpenCV is found); ELSED rows come from the authors' source built separately (Apache-2.0) and are ingested with --elsed-dir in the same file format as the EDLines runner.
Measured 2026-07 on i7-8700K. One caveat repeated from the fairness notes: LSD's official implementation is single-scale and unoptimised C — its 247 ms is the price of its a-contrario validation, not of carelessness; on the quality axis it remains an excellent detector (see the quality page).
